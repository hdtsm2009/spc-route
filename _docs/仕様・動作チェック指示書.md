## A. 仕様チェック（設計どおりか）

### A-1. ランクの意味（最重要）
掲載状況でランクが決まるのが今回の核心です。候補一覧やルート表でランクの定義を確認します。

| ランク | 意味 | 確認 |
|---|---|---|
| **S** | スポカフェ＋ファンスタ **両方掲載**（最優先） | [ ] |
| **AS** | **スポカフェのみ掲載**（無料→有料転換／上位アップセルの本命） | [ ] |
| **AF** | **ファンスタのみ掲載**（競合のみ掲載→スポカフェへ奪取） | [ ] |
| **B** | 両方未掲載だが有望（スポーツバー・観戦設備あり 等） | [ ] |
| **C** | 両方未掲載・温度低め | [ ] |

- [ ] 並び順が「ランク（S→AS→AF→B→C）→スコア→近い順」になっている
- [ ] **AS が AF より上**に並ぶ（同条件のとき）
- [ ] 「仕組みと選定基準」ページ（guide.html）のランク表も上の定義と一致している

### A-2. ピッチ（ひとことの売り文句）
候補カード／ルート表に各店の訪問動機が出ます。掲載状況に応じて以下が出るか。

- [ ] 両方掲載 → 「両サービス掲載◎」
- [ ] スポカフェ無料掲載 → 「スポカフェ無料→有料転換」
- [ ] スポカフェ有料掲載 → 「スポカフェ◯◯→上位」（◯◯にプラン名）
- [ ] ファンスタのみ → 「ファンスタのみ→奪取」
- [ ] 未掲載 → 「スポーツバー◎(未掲載)」「観戦設備あり(未掲載)」「評価◯★(未掲載)」等

### A-3. 概算位置バッジ（番地なし掲載店）
番地が無い掲載店を「町名の中心点」で概算表示しています。誤誘導防止の明示が要点。

- [ ] 概算の店に「概算位置(町名)」バッジが付いている
- [ ] バッジにマウスを乗せる/タップで「番地が無いため町名の中心点。現地で要確認」と分かる
- [ ] 概算の店は実在の番地ではなく町名中心であることが、見て分かる

### A-4. 候補の抽出条件
- [ ] 起点からの半径内（既定 約1,200m）の店だけが出る
- [ ] 「訪問NG」「スペース貸し業態」などの除外店が候補に出ていない
- [ ] 営業時間外が明確な店は出ない／不明な店は「🟡 要確認」で残る

### A-5. ルート最適化
- [ ] 選んだ店が地理的に近い順（一筆書きに近い順）に並ぶ
- [ ] 「起点に戻る」を指定すると最後に起点へ戻る順番になる
- [ ] 行ったり来たり（明らかな交差）が起きていない

## B. 動作チェック（実際に使えるか）

### B-1. 起点の指定（3通り）
- [ ] エリアチップ（西日暮里・難波・渋谷 等）を押すと候補が出る
- [ ] 駅名／住所の自由入力（例「新宿駅」）で候補が出る
- [ ] 店名入力でその店を中心に候補が出る

### B-2. 候補の選択・絞り込み
- [ ] 上位の店が初期で数件チェック済みになっている
- [ ] カードをタップで選択ON/OFFが切り替わる
- [ ] ランクフィルタの **S / AS / AF / B / C** チップがそれぞれON/OFFで効く
- [ ] 「スコア順／近い順」の並び替えが効く

### B-3. ルート発行
- [ ] 「行く店」を選んでルートを発行できる
- [ ] 各店に到着時刻・徒歩移動・滞在時間の目安が出る
- [ ] Googleマップ用のルートリンクが開ける（実際にマップが立ち上がる）
- [ ] 各店の「🗺地図」「📞電話」「🌐HP」リンクが正しく機能する

### B-4. アポ起点モード
- [ ] アポ（確定訪問先）を指定でき、📌で固定表示される
- [ ] アポに時刻を入れると、前後の空き時間に店が配置される
- [ ] 開始〜終了の活動時間内に収まる件数になっている

### B-5. 周辺・表示
- [ ] 「仕組みと選定基準」ページへのリンクが開く
- [ ] スマホでレイアウトが崩れない（横スクロール多発・文字はみ出しが無い）
- [ ] 一覧の件数表示が選択・絞り込みに連動して更新される

## C. 重点的に見てほしい観点（営業の実務として）
- [ ] **AS（スポカフェ無料掲載店）が上位に来ているか** … 今回の営業の本丸。ここが弱いと意味が無い
- [ ] 概算位置の店を「正確な住所」と勘違いしない作りになっているか
- [ ] 1日で実際に回れる件数・距離感になっているか（多すぎ／遠すぎないか）
- [ ] ランク・ピッチを見て「次にどの店へ行くべきか」が直感的に分かるか
